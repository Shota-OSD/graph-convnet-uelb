# SeqFlowRL: Approximation Ratio が 100% を超える原因分析

**作成日**: 2026-04-17
**対象コード**: `src/seq_flow_rl/training/a2c_strategy.py`, `src/seq_flow_rl/algorithms/sequential_rollout.py`
**観測例**: `Epoch 2/30 | Approx Ratio: 100.57%`

---

## 背景

Approximation Ratio は「モデル解が最適解にどれだけ近いか」を表す指標で、**負荷率は小さいほど良い**ため、式は以下のように定義されている。

```python
# a2c_strategy.py:314
approximation_ratio = (mean_gt_lf / mean_model_lf) * 100
```

コメント上の定義:
- `100%` … モデル解が最適と一致
- `< 100%` … 最適より悪い
- `> 100%` … **理論上あり得ない**（最適解より良い解を見つけた）

しかし実際には `100.57%` のようにわずかに 100% を超えるログが観測される。本ドキュメントではその原因を分析する。

---

## 結論（先に）

主因は **モデル側の `mean_model_lf` が "真の負荷率" よりも過小に計算されていること** である。
特に次の2点が効いている:

1. **バグ**: `_update_edge_usage` が **src → 最初の中間ノードのエッジを edge_usage に積んでいない**（最も有力）
2. **設計上の問題**: `mean_gt_lf / mean_model_lf` が「バッチ平均の比」であり、「サンプルごとの比の平均」ではないため、バッチ内分散が大きい場合に 100% をまたぐ偏りが出る

---

## 原因1: `_update_edge_usage` が先頭エッジを積んでいない（バグ）

### 該当箇所

**パス生成** (`sequential_rollout.py:208-269`):

```python
# L208-209: パス配列は空で初期化（src は入らない）
paths = [[] for _ in range(batch_size)]

# L215: current_nodes に src をセット
current_nodes = src_nodes.clone()  # [B]

# L268: サンプリング結果のみを path に追加
for b in range(batch_size):
    if not reached_dst[b]:
        next_node = next_nodes[b].item()
        paths[b].append(next_node)   # ← src は append されない
```

**需要量の積算** (`sequential_rollout.py:355-363`):

```python
for i in range(len(path) - 1):
    u = path[i]
    v = path[i + 1]
    state['x_edges_usage'][b, u, v] += demand
```

### 何が起きているか

コモディティ `c` の src = `s`、到達先 = `d`、サンプリングで `s → v1 → v2 → d` という経路が生成された場合:

| 実際に使われるエッジ | `_update_edge_usage` で積まれる？ |
|---|---|
| `s → v1` | **× 抜け落ちる**（path に `s` が入っていない） |
| `v1 → v2` | ○ |
| `v2 → d` | ○ |

→ **各コモディティにつき 1 エッジ分、需要が `edge_usage` に反映されない**。

### 影響

- `edge_usage` が過小評価される → `load_factor = max(edge_usage / capacity)` も過小評価
- モデル側の `mean_model_lf` が下がり、`gt / model` の比が **100% を超えうる**
- `max_edge_load` を更新していたエッジがたまたま先頭エッジだった場合、誤差がさらに拡大する

### 修正案

パスの起点を明示的に記録する:

```python
# sequential_rollout.py:208-209 近辺
# src をパスの先頭に入れる
paths = [[src_nodes[b].item()] for b in range(batch_size)]
```

または `_update_edge_usage` 側で `src` を別途考慮する。前者の方が既存コードへの影響が少ない。

ただしこの修正は **train_step の報酬計算**・**approximation_ratio 計算** 両方に影響するため、挙動が変わる範囲を確認したうえで適用する必要がある。

## 原因2: 平均の取り方の偏り（設計上の問題）

### 該当箇所 (`a2c_strategy.py:295-316`)

```python
if isinstance(gt_load_factors, torch.Tensor):
    mean_gt_lf = gt_load_factors.mean().item()
else:
    mean_gt_lf = float(gt_load_factors.mean())

mean_model_lf = load_factors.mean().item()

if mean_model_lf > 0:
    approximation_ratio = (mean_gt_lf / mean_model_lf) * 100
```

### 問題点

これは **「バッチ平均の比」** である:

```
Approx = mean(gt) / mean(model)
```

本来必要なのは **「サンプルごとの比の平均」**:

```
Approx = mean(gt_i / model_i)
```

両者は一般に一致しない（Jensen の不等式、調和平均 vs 算術平均の関係）。

### 具体例

バッチに2サンプル、GT=[0.2, 0.6]、Model=[0.25, 0.4] の場合:

| 指標 | 計算 | 値 |
|---|---|---|
| バッチ平均の比（現行） | 0.4 / 0.325 | **123%** |
| サンプル毎の比の平均（正しい） | (0.2/0.25 + 0.6/0.4) / 2 | **115%** |

特に **GT が低いサンプル（モデルが真似しにくい問題）が混ざる** と `mean_gt_lf` が GT の平均を超過的に引き上げ、100% を越えた値になりやすい。

### 修正案

```python
# サンプルごとの比を計算してから平均
valid = (load_factors > 0) & (gt_load_factors > 0)
per_sample_ratio = (gt_load_factors[valid] / load_factors[valid]) * 100
approximation_ratio = per_sample_ratio.mean().item()
```

## 原因3: 不完全パス（未到達）の扱い

`sequential_rollout.py:227-233` では `max_path_length` （seqflowrl_base.json: 20）到達で打ち切られるが、**未到達パスに対するペナルティは報酬側でのみ適用され**、`edge_usage` には「パスの到達したところまでの需要しか積まれない」。

- 到達できなかったコモディティの需要 = 本来は全エッジに負荷がかかる可能性があった
- しかし実装上は「途中まで歩いた分のエッジ」だけが負荷に加算される
- → モデルの負荷率はさらに過小評価される

学習序盤（Epoch 2時点）は到達率が低いため、この効果が顕著に出る。

---

## 原因4（補足）: 報酬側との二重カウント不在

`_compute_rewards` では `load_factor` と `penalty` の両方を報酬に使うが、`_collect_metrics` の `load_factors` は **ペナルティ前の生の負荷率**。よって報酬の妥当性とは別に、approx ratio 計算は純粋な "model load factor vs gt load factor" の比較になっている点は正しい。

一方で、GT はソルバーが **制約をすべて守って出した最適解** なので「到達失敗」や「容量超過」のような状態は含まない。モデル側が不完全解を出していても、`approximation_ratio` はそれを罰しない（load_factor にしか目を向けない）。→ **approx ratio は "パスの到達性" を反映しない指標** であり、これを単独で見ると誤った楽観的評価になる。

---

## 推奨対応

### 短期（バグ修正）

1. **原因1 の修正**: `paths` の初期化を `[[src_nodes[b].item()]]` に変更し、`_update_edge_usage` が先頭エッジも積むようにする
2. 単体テストで「src → v1 → v2」のパスが `edge_usage[src, v1] += demand` を正しく行うか確認

### 中期（指標の正確化）

3. **原因2 の修正**: approximation_ratio をサンプル単位比の平均に変更
4. **原因3 の対処**: `approximation_ratio` を計算する際、**dst に到達しなかったサンプルを除外**（または不完全パス比率を併記）
5. ログに「Complete Rate」や「Invalid Solution Rate」を追加し、approx ratio 単独で評価しない

### 長期（設計）

6. approximation ratio の定義を「同一問題インスタンスでの比」に正規化
7. バリデーション・テストでは、model が全制約を満たした場合のみ approx を計算するゲートを設ける

## 原因A: 逐次ルーティングの構造的制約

### クレジット割り当て（当初仮説）の再評価

「コモディティ0が容量を使い果たしても、後続コモディティが詰まった際のペナルティがコモディティ0に届かない」という当初仮説は **誤りであった**。

現在の実装では、報酬はすべてのルーティング完了後に計算した1スカラーを全コモディティにブロードキャストしている（`a2c_strategy.py:244-245`）。

```python
rewards_expanded = rewards.unsqueeze(1).expand(-1, num_commodities)  # [B, C]
actor_loss = -(log_probs * advantages).mean()
```

コモディティ0が容量を無駄遣いして後続コモディティが詰まれば `reward` が下がり、コモディティ0の `log_prob` にも **正しく** 負の勾配が流れる。期待値としてのクレジット割り当ては機能している。

### 構造的制約の正確な理解

MIPソルバーは全コモディティを**同時に**考慮して最適なフロー分配を求める。SeqFlowRLは「コモディティ0のパスを決定 → 容量更新 → コモディティ1のパスを決定 → …」と逐次的に決定するため、**コモディティ0の決定は後から変更できない**という不可逆性が残る。

```
最適解（MIPソルバー）: 全コモディティを同時考慮
  コモディティ0: A→C→E  コモディティ1: A→B→E（負荷を分散）
  → max_load = 0.21

モデル（逐次）: コモディティ0決定時にコモディティ1の経路は未確定
  コモディティ0: A→C→E（この時点での局所最適）→ 後続が迂回を余儀なくされる
  → max_load = 0.43
```

ただし、`gnn_update_frequency == 'per_commodity'` の設定では、各コモディティのルーティング時に **更新済みの `x_edges_usage` を入力として GNN を再エンコード** する（`sequential_rollout.py:113-120`）。これにより GNN が「前のコモディティが使ったエッジ」を暗黙的に考慮し、**ある程度の先読みを学習できる余地がある**。そのため「原理的に最適解に到達できない」とは断言できない。

### 定量的根拠と限界

ログ（`seqflowrl_training_20260419_201933.txt`）の観測値：
- Approx% ≈ 48% が30エポック通じて **改善しない**
- Epoch 1 の時点で Comp% は既に 94.7%（到達シグナルは1エポックで収束）
- Approx% だけが頭打ち

この停滞は構造的制約の表れである可能性が高いが、**モデル容量不足・報酬シグナルの粗さ**も並存する原因として否定できない（後述の原因B・C参照）。

### 改善案

1. **コモディティ処理順のランダム化**: 特定コモディティが常に有利な順番になることを防ぎ、汎化性を高める
2. **需要の大きいコモディティを優先**: 大きなフローを先にルーティングして残余容量を小需要コモディティに回す
3. **アーキテクチャ変更**: 逐次決定ではなく反復改善型（初期解を生成 → 繰り返し修正）に移行する

## 原因B: モデル容量不足

### 仮説

GNNエンコーダまたはPolicyHeadの表現力が不足していて、「他のコモディティが使いそうなエッジを避ける」という複雑な空間的推論を学習できていない可能性。

### 判断できる指標

| 指標 | 計測方法 | 解釈 |
|---|---|---|
| **訓練データでの Approx%** | `mode='eval'` を train データに適用 | val/test と同程度なら過小適合（容量不足）、train が大きく良ければ過学習 |
| **隠れ層次元を増やした実験** | `hidden_dim` を 2× に増やして再学習 | Approx% が改善するなら容量が律速していた |
| **勾配ノルムの推移** | `grad_norm = sum(p.grad.norm() for p in model.parameters())` をログ | 学習序盤から小さい・横ばいなら表現力が飽和している |
| **パラメータ数 vs 問題規模** | モデルの総パラメータ数 / グラフのノード×コモディティ数 | 経験則として 1問題インスタンスあたり数百パラメータを下回ると不足気味 |
| **エントロピーの推移** | `mean_entropy` がすぐ 0 に収束しているか | 早期に低下 → 探索不足・表現力限界で局所解に落ちている |

## 原因C: 報酬シグナルの粗さ

### 仮説

現在の報酬 `2.0 - 2.0 * load_factor` はサンプル単位の1スカラーであり、「どのエッジ選択が load_factor を悪化させたか」の情報が含まれていない。load_factor がわずかに違っても同程度の報酬になるため、fine-grained な改善を学習するシグナルが弱い。

### 判断できる 指標

| 指標 | 計測方法 | 解釈 |
|---|---|---|
| **Advantage の分散** | `advantages.var()` をログ | 分散が極端に小さい（< 0.01）→ 報酬差異が小さく勾配が意味を持たない |
| **報酬の値域と分布** | `rewards.min() / .max() / .std()` をログ | 大半のサンプルが同じ報酬帯に集中 → シグナルが粗い証拠 |
| **差分報酬との比較実験** | `reward_c = -(lf_after_c - lf_before_c)` に切り替えて再学習 | Approx% の改善速度が上がれば報酬粗さが律速していた |
| **エポック間の Approx% 変動幅** | 各エポックのApprox%の標準偏差 | 変動が極小（< 0.5%）で停滞 → シグナルが薄くて重みが動いていない |

---

## 原因B・C の切り分け方

1. まず **訓練 Approx% vs 検証 Approx%** を比較する
   - 両方低い → 過小適合（容量不足か報酬粗さ）
   - 訓練は高く検証が低い → 過学習（容量は足りているが汎化に問題）
2. **`hidden_dim` 2× 実験** と **差分報酬実験** を片方ずつ行い、改善した方が律速要因

## 限界1: 逐次決定の構造的制約（主因）

コモディティを1つずつ順番に確定するため、**コモディティ0の選択時点で後続コモディティの需要が考慮できない**という不可逆性がある。

`per_commodity` 再エンコードで「前のコモディティが使ったエッジ」は入力に含まれるが、「後のコモディティが何を必要とするか」は入力に存在しない。MIPソルバーが全コモディティを同時考慮して出す最適解との差は、学習量やモデル容量を増やしても縮まらないことが実験で確認された。

```
MIPソルバー（同時考慮）:
  コモディティ0: A→C→E  コモディティ1: A→B→E（負荷を分散）
  → max_load = 0.21

SeqFlowRL（逐次）:
  コモディティ0: A→B→E（この時点での局所最適）
  コモディティ1: A→B→E は混雑 → 迂回を余儀なくされる
  → max_load = 0.43
```

## 限界2: 固定処理順による偏り

常に `c=0, 1, 2, 3, 4` の固定順で処理するため、コモディティ0は常に空いているグラフで経路を選べる一方、コモディティ4は残り容量の中から選ばざるを得ない。モデルが「順番ごとの振る舞い」に過適合し、推論時の性能が不均一になる。

## 限界3: グローバル価値関数の粗さ

ValueHead はサンプル全体の1スカラー `V(s)` を出力し、コモディティ0〜4の全員が同じ baseline を使って advantage を計算する。「このコモディティの選択が全体にどう貢献したか」という個別シグナルが存在しないため、勾配の分散が大きく、細かい改善を学習するのが遅くなる。

## 限界4: 完全到達できないサンプルが約24%残る

CompSample% が ~76% にとどまり、約4サンプルに1つは全コモディティが宛先に到達できていない。Approx% は到達完了サンプルのみで計算されるため、**実際の解品質はさらに低い**。

### 限界の関係図

```
根本原因
└── 逐次決定（不可逆性）
      ├── 後続コモディティの影響を先読みできない
      │     → Approx% が ~48% で頭打ち（学習・容量増加では解決不可）
      └── 固定処理順の偏り
            → 特定コモディティへの過適合

付随する問題
├── グローバル価値関数
│     → クレジット割り当てが粗い（学習収束を遅らせる）
└── 未到達サンプルの存在（~24%）
      → Approx% が実態より楽観的に見える
```

### 改善の方向性

| アプローチ | 期待効果 | コスト |
|---|---|---|
| 処理順ランダム化 | 限界2の解消、5〜10%改善の可能性 | 低 |
| 差分報酬 | 限界3の緩和、収束速度改善 | 低 |
| 反復改善型（Local Search） | 限界1を根本回避、大幅改善の可能性 | 高 |
| 全コモディティ同時処理（Transformer等） | 限界1を根本回避 | 非常に高 |

現アーキテクチャのままで Approx% を 60〜70% 台に乗せるには処理順ランダム化＋差分報酬の組み合わせが現実的な上限。80% 以上を目指すなら反復改善型への移行が必要。
